Last two things

Inline editing scope + what to do with existing pages.

D. Which widgets get inline (click-and-type) editing?

D1

Minimal: Heading, Text Editor, Button label only

The three obvious ones. Everything else (Testimonial quote, Pricing features, Alert message, Icon Box title/text) you edit via the style panel's Content tab using normal form fields. Faster to build.

D2

Full: every text-bearing widget supports inline edit

Heading, Text, Button, Icon Box title/desc, Testimonial quote/author, Pricing title/price/feature rows, Alert message. More work (each widget needs editable-region markup + RTE wiring) but feels much more Elementor-like.

Recommendation: D2. Inline editing is one of the main reasons people choose Elementor over GrapesJS. Worth the extra ~3 days.

E. What happens to pages that already exist?

Your seeders create a few placeholder pages. The existing 4 custom blocks output Tailwind-class HTML — that HTML is stored in builder_page_translations.html. With Option 2 (continuous CSS style panel) and a new Section/Column primitive, those old pages don't match the new model.

E1

Reset seed data — rebuild seeded pages with new widgets

Update seeders to author the Welcome / placeholder pages using the new Section/Column/Widget JSON. Run migrate:fresh --seed and you're done. Best for early-stage projects — you don't have real content yet. Simplest path.

E2

Coexist — old pages render as legacy HTML, edit in new builder when ready

Existing translations with raw HTML keep rendering as-is via the public PageController (the rendered output doesn't change). When you open such a page in the new editor, it loads into a single "Legacy HTML" wrapper widget. You can leave it or rebuild section-by-section. Best if you have real published content you don't want to break.

E3

Auto-migrate old HTML → new component tree

One-time migration parses each translation's HTML and converts it into a best-effort Section/Column/Widget tree (Hero block → Section with Heading + Text + Button widgets, etc.). Brittle — works for the 4 known blocks, fails for arbitrary content. Significant extra work for limited return.

Recommendation: E1. The project is young, seeded pages are placeholders, you'll get a clean slate. If you've published real content to clients, switch to E2.
