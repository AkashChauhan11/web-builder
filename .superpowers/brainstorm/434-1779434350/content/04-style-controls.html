How should the style controls work?

Your existing blocks use Tailwind utility classes. Elementor uses continuous CSS values. This decision shapes the entire style panel UI and what gets saved.

1

Tailwind preset controls (recommended)

Pro: Output is pure Tailwind classes — consistent with existing blocks, no <style> blocks needed, fast pages. Per-breakpoint “just works” (md:py-12). UI is faster to build (no sliders, no unit logic).
Con: Discrete values only (can't pick 27px padding). For most marketing pages this is fine; designers may want more.

2

Elementor-style continuous values

Pro: Pixel-perfect control. Truly feels like Elementor.
Con: Output is inline style="" attributes that fight Tailwind specificity. Per-breakpoint requires <style> blocks with media queries. Existing Tailwind blocks would need to either be migrated to inline CSS or coexist messily. ~2× the implementation work for the style panel.

3

Hybrid — Tailwind presets + arbitrary values

Default UI shows preset chips (sm/md/lg). A small “custom” toggle next to each control lets you type any value — emits Tailwind arbitrary value py-[27px] instead of inline CSS. Continuous when you need it, clean utility classes otherwise. Most flexible.
Con: A bit more UI to build than Option 1. But arbitrary values stay in the Tailwind world — no inline style="" attributes, no mixed model.

Strong recommendation: Option 3. You get the continuous control Elementor users expect, the output stays pure Tailwind (no <style> blocks, no specificity wars), and the existing Hero/CTA/etc. blocks keep working unchanged.
